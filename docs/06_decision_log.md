# Decision Log

记录项目开发过程中的重要技术决策。

重点记录：

为什么这样设计。

而不仅记录是什么。

---

## 决策 001

采用 Python + pandas 作为基础框架。

### 背景

项目核心是时间序列仿真。

### 原因

pandas 的：

* DataFrame；
* Series；
* DatetimeIndex；

非常适合时序数据。

学习成本较低。

开发效率较高。

### 放弃方案

* Excel；
* MATLAB；
* 数据库驱动；
* 复杂框架。

### 当前结论

继续采用：

Python + pandas。

---

## 决策 002

采用面向对象方式组织项目。

### 背景

随着功能增加，

单文件脚本会快速失控。

### 原因

每个对象对应一个 Model：

* Weather；
* PV；
* Load；
* Battery；
* EMS。

符合实际工程认知。

### 放弃方案

* 全部函数化；
* 单文件开发。

### 当前结论

采用：

一个对象对应一个 Model。

---

## 决策 003

组合优于继承。

### 背景

项目规模有限。

### 原因

复杂继承结构收益较低。

容易增加理解成本。

### 放弃方案

* 基类；
* 父类；
* 复杂继承体系。

### 当前结论

采用：

多个独立 Model。

通过组合形成系统。

---

## 决策 004

Model 负责计算。

测试文件负责组织数据。

### 背景

需要保持职责清晰。

### 原因

避免算法和测试逻辑耦合。

### 放弃方案

在测试文件中编写算法。

### 当前结论

Model：

负责计算。

Test：

负责组织数据。

---

## 决策 005

正确性优先。

简单性优先。

工程实用优先。

### 背景

项目定位是工程分析工具。

### 原因

结果正确比代码优雅更重要。

### 放弃方案

过度追求复杂设计。

### 当前结论

保持：

正确性优先；

简单性优先；

工程实用优先。

---

## 决策 006

避免复杂设计模式。

保持渐进式演化。

### 背景

项目规模仍然有限。

### 原因

复杂架构收益有限。

### 放弃方案

* 工厂模式；
* 观察者模式；
* 复杂接口层；
* 过度抽象。

### 当前结论

保持简单。

逐步演化。

## 决策 007

不同模块之间需要统一约定。

### 当前结论

↓

上下层结构。

季节、

电价、

尖峰、

* 上网电价；

## 决策 013

EMS 采用多目标架构。

### 背景

随着功能增加，

单一模式切换逐渐失效。

### 原因

多个目标可能同时成立。

例如：

* 光伏消纳；
* 峰谷套利；
* 需量控制。

### 放弃方案

Single Mode。

### 当前结论

采用：

Multi-objective。

多个目标允许同时激活。

---

## 决策 014

采用优先级架构。

### 背景

不同目标的重要程度不同。

### 当前结论

目标优先级：

Constraint Protection

↓

Loss Reduction

↓

Profit Enhancement

↓

Secondary Objective

其中：

Constraint Protection：

负责：

* PCS限制；
* SOC限制；
* 变压器限制；
* 反送电限制。

Loss Reduction：

负责：

* 减少弃光；
* 需量管理。

Profit Enhancement：

负责：

* 峰谷套利。

Secondary Objective：

负责：

* 均衡循环；
* Reserve；
* 其他目标。

---

## 决策 015

所有价值流都需要进行经济性比较。

### 背景

储能循环存在寿命成本。

### 原因

储能动作并不天然具有经济价值。

### 当前结论

所有价值流：

* 光伏消纳；
* 峰谷套利；
* 需量控制；

都需要比较：

收益

vs

循环寿命成本。

只有：

收益

>

循环成本

时，

才具有经济价值。

## 决策 016

建立 DataModel 作为统一数据层。

### 背景

随着测试数据、气象数据和未来外部数据源增加，

不同数据格式逐渐增加。

WeatherModel 同时承担：

数据读取

+

统计分析

职责逐渐变重。

### 原因

数据处理和业务算法应当解耦。

避免：

WeatherModel 成为上帝对象。

### 放弃方案

由各个 Model 自行读取 csv。

### 当前结论

增加：

DataModel

负责：

* csv读取；
* datetime转换；
* DatetimeIndex建立；
* 列名标准化。

输出：

标准 DataFrame。

形成：

Data Layer

↓

Business Layer

↓

Result Layer

三层结构。

Business Model

不再关心：

数据来源。

## 决策 017

采用 main.py 作为 Golden Path。

### 背景

项目由单人开发。

模块数量逐渐增加。

继续以大量 test_xxx.py 作为主要验证方式，

会增加维护成本。

### 原因

系统可运行性

比

测试覆盖率

更重要。

### 放弃方案

以 Unit Test 为主要开发入口。

### 当前结论

python main.py

作为：

* 开发入口；
* 集成测试入口；
* 发布入口。

开发流程：

修改一个模块

↓

python main.py

↓

分析：

output/result.csv

↓

分析：

output/*.png

↓

Debug

↓

git

↓

更新 docs

↓

进入下一功能。

采用：

Golden Path 开发模式。

始终保持系统可运行。
