# Decision Log

记录项目开发过程中的重要技术决策。

重点记录：

为什么这样设计。

而不仅记录是什么。

---

# 2025~2026

## 决策 001

采用 Python + pandas 作为基础框架

### 原因

项目核心是时间序列仿真。

pandas 的：

- DataFrame；
- Series；
- DatetimeIndex；

非常适合时序数据。

学习成本较低。

开发效率较高。

### 放弃方案

Excel；

MATLAB；

数据库驱动；

复杂框架。

### 当前结论

继续采用：

Python + pandas。

---

# 决策 002

采用面向对象方式组织项目

### 原因

随着功能增加，

单文件脚本会快速失控。

每个对象对应一个类：

- Weather；
- PV；
- Load；
- Battery；
- EMS；

符合实际工程认知。

### 放弃方案

全部函数化；

单文件开发。

### 当前结论

采用：

一个对象对应一个 Model。

---

# 决策 003

组合优于继承

### 原因

项目规模有限。

复杂继承结构收益较低。

容易增加理解成本。

### 放弃方案

基类；

父类；

复杂继承体系。

### 当前结论

采用：

多个独立 Model。

通过 main.py 进行组合。

---

# 决策 004

main.py 负责调度

### 原因

main.py 应作为：

系统入口。

负责：

调用顺序。

而不负责：

具体算法。

### 当前结构

main.py

↓

WeatherModel

↓

PVModel

↓

LoadModel

↓

BatteryModel

↓

ResultModel

---

# 决策 005

算法放入 Model 内部

### 原因

保持职责清晰。

方便后续扩展。

### 当前结论

main.py：

负责调度。

Model：

负责计算。

---

# 决策 006

优先保证正确性

### 原因

项目主要用于工程分析。

结果正确比代码优雅更重要。

### 当前原则

正确性优先。

简单优先。

工程实用优先。

### 原因

当前项目规模较小。

复杂设计模式收益有限。

### 放弃方案

# 决策 009

逐步演化，而不是一次设计完成

### 原因

需求会持续变化。

很多结构需要在实践中形成。

### 当前原则

先运行。

再优化。

避免提前设计。

---

# 决策 010

采用 Debug → Freeze 开发方式

### Debug 阶段

允许：

print；

临时变量；

调试输出。

### Freeze 阶段

算法验证完成后：

停止修改。

减少输出。

进入稳定状态。

### 当前状态

WeatherModel：

Freeze。

BatteryModel：

接近 Freeze。

---

# 决策 011

统一正负号定义

### Power Convention

PV Power：

正。

Load Power：

正。

Battery Power：

正：

放电。

负：

充电。

Grid Power：

正：

购电。

负：

上网。

### 原因

保证整个系统统一。

避免后续模型混乱。

### 当前状态

已经形成统一约定。

适用于整个项目。

---

# 决策 012

BatteryModel 不负责控制策略

### BatteryModel 职责

负责：

- 功率限制；
- SOC限制；
- 效率；
- SOC更新；
- Grid计算。

### 不负责

什么时候充；

什么时候放。

### 原因

控制策略属于 EMS。

### 当前结论

未来：

EMSModel

↓

BatteryModel

形成上下层结构。

↓

↓

调试

↓

Freeze

继续坚持。
